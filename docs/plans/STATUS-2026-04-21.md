# SabaiApply — Status Report

**Date:** 2026-04-21
**Accelerator interview:** ~2026-04-24 (Zero to 1 @ KMITL / CMKL)
**Current branch:** `student-auth`

This document is a grounded "what's built, what's left" view of the product, split by **Student Side** and **University Side**. It's based on the two plans in `docs/plans/` (Claude Progress Report + Complete Strategy & Product Document) and a direct audit of the codebase.

---

## 1. Product scope in one line

Students build one profile and apply to any Thai university from a single dashboard. Universities receive standardized, pre-screened applications in a faculty-scoped review dashboard. MVP focus is **TCAS Round 1 (Portfolio)**.

## 3. What's left (the honest gap)

### 🎓 Student Side — gaps

**High priority (original Week 1-2 MVP list, not done)**
- [ ] **Portfolio builder (form-based)** — CLAUDE.md Week 1 Friday. `portfolio_items` exists but there's no "build a sabaiapply portfolio with essay + categorized sections + generate PDF" flow. Activities section is the closest thing but it's not positioned as a portfolio builder.
- [ ] **Portfolio import** — CLAUDE.md Week 2 Monday. TCASFolio URL paste + iFolio/freestyle PDF upload stored with `source` field. Not started.
- [ ] **Eligibility checker (green/yellow/red)** on the student-facing browse view — CLAUDE.md Week 2 Wed. Admin side screens applications; student side doesn't preview their odds before applying. Would block submission if hard requirements not met.

**High priority (original Week 3 list, not done)**
- [ ] **Email notifications via Resend** — submission confirmation, status change, results published. `src/lib/email.ts` helpers don't exist yet. Zero `Resend` usage anywhere in the codebase.
- [ ] **Interview slot booking** — after shortlisted, show available `interview_slots` for program → student picks → update row to `status=booked`. Table exists in schema, no UI.
- [ ] **Results page** — per-app accepted/waitlisted (with position)/rejected with next steps. Currently only shown as a status badge.

**Medium priority**
- [ ] Forgot password on student side (admin side has it; student side does not)
- [ ] Social login (Google) — explicitly out of MVP per CLAUDE.md but students will ask
- [ ] Uni search filters: city/distance UI exists but doesn't filter (placeholder)

**Low priority (post-launch per CLAUDE.md)**
- AI-powered recommendations, student-to-student messaging, payment processing, advanced portfolio templates, English/Thai toggle finesse, push notifications, alumni stats

---

### 🏛️ University Side — gaps

**High priority**
- [ ] **Interview scheduling (admin side)** — faculty creates slots in `interview_slots`, sees booked/available, detects no-shows → auto-notify waitlist. Schema exists, no UI.
- [ ] **Results publishing (one-click)** — bulk set `applications.status` to accepted/waitlisted/rejected + trigger student notifications. Status change is currently per-application only.
- [ ] **Waitlist position tracking + auto-advance** — when a student withdraws or declines, next waitlisted student gets promoted + notified.
- [ ] **Confirmation tracking dashboard** — after results, real-time view of who confirmed vs pending during TCAS 48-hour window.

**Medium priority**
- [ ] **TCAS export** — one-click CSV in CUPT-compatible format for Round 1 reporting.
- [ ] **Bulk actions on applications list** — shortlist/reject/export multiple at once (plan §6 Faculty Features, page 16 PDF).
- [ ] **Multiple reviewers per application** — rubric supports it in schema but UI assumes single scorer.
- [ ] **University-wide / cross-faculty admin view** — central admin sees across all faculties with comparison. Super admin does this for the platform; uni-admin needs a uni-internal equivalent.

**Low priority (post-MVP, explicitly deferred per strategy doc §14)**
- AI chatbot, housing/scholarship database, payment processing (Omise), TCAS full integration, Round 2/3/4 flows, Round 4 marketplace, international admissions premium tier, LINE notifications

## 4. What's been deferred (on purpose — don't build these yet)

Per the Complete Strategy doc §14 "What NOT to do right now":

- Mobile app (web-first)
- AI chatbot (value-add, after Round 1 works)
- Payment processing (universities bill students directly in beta)
- Round 3 advisor mode (free feature, comes after monetization proven)
- Round 4 marketplace (requires pilot data first)
- Housing / scholarship database
- Student-paid premium features

## 5. Recommended priority for the next 2 weeks (accelerator-driven)

**Interview is ~2026-04-24.** What matters for that demo is a working Round 1 Portfolio loop, not breadth. Fix the gaps in this order:

### Before accelerator demo (3 days)
1. **Seed 1 real university end-to-end** — KMITL with 1-2 faculties, real programs, real requirements. Already partly done via `thai-universities.ts`; needs real program_requirements rows.
2. **Create a demo student account** with full profile, all 6 Common App sections completed, 1 portfolio, 1 essay per university.
3. **Walk the full flow on a phone** — signup → profile → add KMITL → fill essay → Review & Submit → see status on dashboard. Fix anything that breaks mid-demo.
4. **Admin side demo** — log in as faculty admin → see the submitted application → green/yellow/red screening → score it → shortlist. This is the "aha" moment for universities.

### Post-accelerator (if we get in)
5. **Portfolio builder** (student Week 1 Fri) — unlocks the full "Round 1" narrative.
6. **Email notifications via Resend** (student Week 3 Mon) — submission confirmations are table-stakes.
7. **Interview booking** (both sides) — closes the Round 1 loop per strategy doc §7 "Round 1 Portfolio — Full Workflow".
8. **Results publishing + confirmation tracking** — the highest-value admin feature per strategy doc §10 ("Problem 5 & 6").

### What NOT to touch before the demo
- Don't start AI chatbot, don't add Round 2/4 flows, don't build payment. All explicitly deferred in the strategy doc and would pull focus.
